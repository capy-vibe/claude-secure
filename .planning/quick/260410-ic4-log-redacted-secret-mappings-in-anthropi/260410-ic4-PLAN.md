Purpose: Currently the proxy logs `redacted: N` (a count) but not WHICH secrets were matched. The operator needs to see pairs like `"gh_4924..." => "<redacted_user_token>"` to verify the right secrets are being caught.

Output: Modified proxy.js that logs a `redaction_map` array with partially-masked cleartext values and their placeholders.
</objective>

  <action>
Modify `buildMaps()` in proxy/proxy.js to return an additional `logPairs` array alongside `redactMap` and `restoreMap`. For each secret that has a matching env var, build a log-safe entry:

```js
{ masked: realValue.slice(0, Math.min(8, Math.floor(realValue.length / 3))) + '...', placeholder: entry.placeholder, env_var: entry.env_var }
```

This shows enough of the cleartext to identify WHICH secret was matched (e.g., `"gh_49247..."`) without leaking the full value into the log file.

Then, in the request handler (the `req.on('end', ...)` callback), after `buildMaps()` is called and before the request is forwarded, add a log entry when there are active redaction mappings:

```js
if (logPairs.length > 0) {
  logJson('info', 'Active redaction mappings', { redaction_map: logPairs });
}
```

Update the destructuring of `buildMaps()` return value from `{ redactMap, restoreMap }` to `{ redactMap, restoreMap, logPairs }`.

Do NOT log full secret values. The masked prefix approach (first 8 chars or 1/3 of length, whichever is smaller) is intentional -- enough to identify the secret, not enough to use it.
