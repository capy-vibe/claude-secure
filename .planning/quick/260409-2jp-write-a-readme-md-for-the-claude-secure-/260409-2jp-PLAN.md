**Structure:**

1. **Title and tagline** -- "claude-secure" with a one-line description: security wrapper for Claude Code that prevents secret exfiltration via Docker network isolation.

2. **Problem statement** -- 2-3 sentences explaining why this exists: Claude Code has full network access, reads .env files, secrets enter LLM context and get sent to Anthropic, tool calls like `Bash(curl ...)` can exfiltrate to arbitrary URLs.

3. **How It Works** -- Describe the four-layer defense-in-depth architecture:
   - Layer 1: Docker network isolation (`internal: true` network blocks all external access)
   - Layer 2: PreToolUse hook validates every Bash/WebFetch/WebSearch call, checks domains against whitelist, blocks payloads to non-whitelisted domains, detects obfuscation attempts
   - Layer 3: Anthropic proxy intercepts all Claude-to-Anthropic traffic, redacts secret values, replaces with placeholders, restores on response
   - Layer 4: Call validator with SQLite registration and iptables enforcement -- hook registers call-IDs, validator manages network rules, call-IDs are single-use with 10s TTL

   Include an ASCII diagram showing the three containers (claude, proxy, validator), the two networks (internal, external), and the flow of traffic. The proxy sits on both networks. The validator shares the claude container's network namespace (`network_mode: service:claude`).

4. **Prerequisites** -- Docker, Docker Compose v2, curl, jq, uuidgen. Linux (native) or WSL2. No macOS support.

5. **Installation** -- Show the commands:
   ```
   git clone <repo-url>
   cd claude-secure
   ./install.sh
   ```
   Briefly describe what the installer does: checks dependencies, detects platform (Linux/WSL2), prompts for auth (OAuth token recommended, API key fallback), sets workspace path, copies files to ~/.claude-secure, builds Docker images, installs CLI.

6. **Usage** -- Show the CLI commands from bin/claude-secure:
   - `claude-secure` -- start containers and launch Claude Code
   - `claude-secure stop` -- stop containers
   - `claude-secure status` -- show container status
   - `claude-secure update` -- pull latest and rebuild

7. **Configuration** -- Explain whitelist.json with the actual default structure from config/whitelist.json. Show:
   - `secrets` array: each entry has `placeholder`, `env_var`, `allowed_domains`
   - `readonly_domains` array: domains allowed for GET requests without registration
   - Explain that the whitelist is read fresh on every request (no restart needed)
   - Note that the user's copy lives at `~/.claude-secure/whitelist.json` (symlinked by installer)

8. **Architecture Details** -- Brief subsections for each container:
   - `claude` container: Ubuntu-based, runs Claude Code, all capabilities dropped, no-new-privileges, workspace volume mounted
   - `proxy` container: Node.js, stdlib only (no npm dependencies), buffered request/response, bidirectional secret redaction
   - `validator` container: Python, stdlib only (no pip dependencies), SQLite WAL mode, iptables with NET_ADMIN capability

9. **Security Model** -- Summarize the guarantees:
   - Secrets in LLM context are redacted before reaching Anthropic
   - Outbound payload calls require whitelisted domain + registered call-ID
   - Read-only (GET) requests allowed without registration
   - Shell obfuscation (variable expansion, eval, base64 in curl commands) is detected and blocked
   - Hook scripts and config are root-owned and read-only inside the container
   - Zero external dependencies in proxy and validator (stdlib only) to minimize supply-chain risk

10. **Limitations / Known Gaps** -- Be honest:
    - Buffered proxy (no streaming) -- responses are slower
    - No `@file` content scanning -- secrets in files Claude references via `@file` are not caught
    - No macOS support
    - No automatic OAuth token refresh

11. **Platform Support** -- Linux (native) and WSL2. Note that Docker Desktop on WSL2 may have iptables issues; Docker CE directly in WSL2 is recommended.
